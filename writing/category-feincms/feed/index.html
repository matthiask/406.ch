This does not have to be a bad thing &ndash; FeinCMS is compatible with the recently released Django 2.1 back to Django 1.7, it works well and it&rsquo;s pain points are generally well known by long time users. Slow development also means minimal breakage because of new incompatibilities, not bad either.</p> <p>That being said, here&rsquo;s a list of things with I find bothersome when working with FeinCMS:</p> <ul> <li>The extension mechanism and <code>create_content_type</code> are brittle. Many hacks and workarounds and fumbling with Django&rsquo;s internals were necessary over the years to avoid invalid and/or incomplete model meta information. Also, testing the dynamic creation of models is hard to test and most breakages were very hard to reproduce.</li> <li>It has a few features which make it hard to know what&rsquo;s going on. There&rsquo;s no single place to look if you want to know what fields the page model has. There&rsquo;s no place no look if you want to know what happens when processing a request because request and response processors and also the content types themselves all have ways to interfere and even short circuit request processing.</li> <li>The implicitness of the automatic registration of content type inlines with the administration interface made it necessary to invent new hacks (<code>feincms_item_editor_form</code> and <code>feincms_item_editor_inline</code>) to still be able to customize the inline class.</li> <li>Models and rendering is coupled too closely. Adding a <code>render()</code> method to content type models works well enough for small-to-medium sized websites, but falls apart quickly for infrastructure projects. The coupling itself does not prevent people from implementing better and more decoupled solutions, but doing the ugly thing is just too easy.</li> <li>The <code>ApplicationContent</code> way of integrating apps was nice, but mostly you wouldn&rsquo;t want more than one app per page anyway. The complexity cost of this flexibility e.g. for running apps (calling the resolver ourselves) and for reversing URLs (impossible to use Django&rsquo;s <code>reverse()</code> function directly) is higher than necessary.</li> <li>To be honest I don&rsquo;t understand the <code>feincms.extensions.translations</code> extension and its redirect mechanisms anymore.</li> <li>The reusability of the base model and the page view is enormously bad. Reimplementing the view by calling all processors and content types&rsquo; <code>process()</code> and <code>finalize()</code> methods and so on is just horrible.</li> </ul> <p>That being said, FeinCMS is used in a few flagship projects which we&rsquo;re still actively developing, which means that FeinCMS won&rsquo;t be going away for years to come.</p> <p>This post is part of a series, <a href="https://406.ch/writing/the-future-of-feincms/">The future of FeinCMS</a>.</p>The other future of FeinCMS: django-content-editor and feincms3https://406.ch/writing/the-other-future-of-feincms-django-content-editor-and-feincms3/2018-09-22T12:00:00+00:002018-09-22T12:00:00+00:00<h1>The other future of FeinCMS: django-content-editor and feincms3</h1> <p>The FeinCMS ItemEditor code lives on in a new project, <a href="https://django-content-editor.readthedocs.io/">django-content-editor</a><sup>1</sup>.</p> <p>By contrast <a href="https://feincms3.readthedocs.io/">feincms3</a> is a completely new project, which was the result of starting to build websites with a stripped down environment of django-content-editor and django-mptt<sup>2</sup> only.</p> <p>feincms3 avoids many of the downsides of FeinCMS 1.x by doing less. At first sight it sounds like building a website with feincms3 will be more work. In my view this does not have to be, since feincms3 offers more direct ways of achieving the same results.</p> <p>feincms3 addresses <a href="https://406.ch/writing/the-state-of-things-in-feincms-1x-land/">FeinCMS&rsquo; issues</a> by:</p> <ul> <li>It does not offer concrete models, only mixins. Because of this, all problems with dynamic model creation and the extensions mechanism simply disappear.</li> <li>You write the page model yourself, so you&rsquo;ll know what fields it has.</li> <li>No request and response processors, and plugins (content types in FeinCMS 1.x) do not have any ways of interferring with request and response processing. This makes at all times obvious what&rsquo;s going to happen when processing a request. Redirects? Easy, DIY in your own, short view function.</li> <li>The automatic creation of inlines for the admin interface is gone. This avoids the custom <code>feincms_item_editor_inline</code> protocol, and its coupling of models and the administration interface.</li> <li>Models do not have a render method. Instead, there&rsquo;s a <a href="https://feincms3.readthedocs.io/en/latest/build-your-cms.html#rendering-and-templates">renderer</a> where you&rsquo;re supposed to register plugins. This makes it straightforward to use different renderers for different purposes, and the preferred way to achieve this is obvious.</li> <li>The new application mixin (replacing FeinCMS 1.x <code>ApplicationContent</code>) works with Django&rsquo;s own <code>reverse()</code> without any monkey patching.</li> <li> <p>And last but not least feincms3 does not only not ship a complicated view function, it does not ship a view function <em>at all</em>. The reason is that <a href="https://feincms3.readthedocs.io/en/latest/build-your-cms.html#views-and-urls">writing feincms3 views is extremely straightforward</a> and mostly requires only a few feincms3-specific lines of code in real-world projects.</p> </li> <li> <p>The TreeEditor has been added to <a href="https://django-mptt.readthedocs.io/en/latest/admin.html#mptt-admin-draggablempttadmin">django-mptt</a> as <code>DraggableMPTTAdmin</code>.</p> </li> <li>We have since moved to <a href="https://github.com/matthiask/django-tree-queries">django-tree-queries</a>, but now&rsquo;s not the time for this story. Search through feincms3&rsquo;s <a href="https://feincms3.readthedocs.io/en/latest/project/changelog.html">CHANGELOG</a> if you want to know more right now.</li> </ul> <p>This post is part of a series, <a href="https://406.ch/writing/the-future-of-feincms/">The future of FeinCMS</a>.</p>The future of FeinCMShttps://406.ch/writing/the-future-of-feincms/2018-09-22T12:00:00+00:002018-09-22T12:00:00+00:00<h1>The future of FeinCMS</h1> <p>A recent exchange in an <a href="https://github.com/feincms/feincms-elephantblog/issues/68">issue in the elephantblog repository on Github asking whether the project was still alive</a> and my own work on <a href="https://feincms3.readthedocs.io">feincms3</a> made me think about the need to clarify my position on the future of <a href="https://github.com/feincms/feincms">FeinCMS 1.x</a>.</p> <p>In the issue I wrote that it (meaning elephantblog) is mostly in maintenance mode and not actively developed anymore, and I also pointed people towards feincms3 for new projects.</p> <p>There are a lot of things FeinCMS 1.x got right and there are things that could be better or I could have handled better.</p> <p>FeinCMS is used in a few flagship projects which we&rsquo;re still actively developing, which means that FeinCMS won&rsquo;t be going away for years to come.
